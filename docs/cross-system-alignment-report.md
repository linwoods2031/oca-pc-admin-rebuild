# 跨系统上线对齐报告

本文用于把 PC 后台、已认可小程序壳、恢复后端和 SQL 热修材料放到同一张发布证据表里。它只记录脱敏的功能、规则和发布闸门，不包含真实账号、真实患者信息、真实 allow-list id 或生产验证细节。

## 当前结论

- PC 后台正式版：一般情况表和当前用药保持只读；修改入口保留在已认可的平板/小程序流程。
- PC 后台量表展示：不在前端硬编码小程序最终量表评分、结论和题目规则，优先展示后端返回的名称、得分和结论。
- 小程序最终口径：以业务用户最后确认的小程序要求为准，记录见 `docs/mini-program-final-requirements.md`。
- 恢复契约：患者归属字段、量表保存 payload、提交态字段、一般情况表和用药存储范围已从恢复后端和已认可小程序壳反查，记录见 `docs/recovered-contracts.md`。
- 本地对齐证据：`npm run release:cross-system:markdown` 会扫描本机小程序维护目录、已认可小程序壳、恢复后端热修源码和 SQL 热修文件，输出跨系统对齐报告。
- 直接正式上线：仍不允许。正式上线前必须由发布负责人补齐运行包一致性、微信正式版发布、备份、回滚、审计、权限和变更窗口证据。

## 如何生成证据

在 PC 后台仓库根目录运行：

```sh
npm run release:evidence:verified:markdown
npm run release:cross-system:markdown
```

如果小程序维护目录或小程序壳目录不在默认位置，使用环境变量指定：

```sh
OCA_MINI_MAINTENANCE_DIR=<mini-maintenance-dir> \
OCA_MINI_SHELL_DIR=<mini-shell-dir> \
npm run release:cross-system:markdown
```

这两个命令的用途不同：

- `release:evidence:verified:markdown`：证明 PC 仓库自身的源码 gate、测试、构建、CI 和正式评审候选状态。
- `release:cross-system:markdown`：证明 PC 口径、小程序最终要求、恢复后端热修和 SQL 热修材料在本机恢复材料中能对齐。

动态输出只作为评审单附件，不提交到仓库。

## 脚本检查项

`scripts/cross-system-alignment.mjs` 会检查：

- PC 源码是否强制一般情况表和当前用药只读。
- PC 文档是否声明小程序最终口径是量表规则来源。
- 小程序维护记录是否区分最新体验版和旧正式长期码。
- 最新体验二维码文件及 `latest` 别名是否存在并一致。
- 小程序壳是否包含住院号搜索、修改患者、上次得分/结论、预计复诊日期、量表全选/清空、居住楼层、电梯房、跌倒史、义齿等最终需求。
- SQL 热修材料是否覆盖 SAS、MMSE、FRA、FRIED、AIS、SPPB、ADL、IADL、中医体质等最终规则。
- 恢复后端热修源码是否覆盖 ADL/IADL/FRA 排序、性别题过滤、MMSE 文化程度、体质辨识性别标记。

脚本只输出文件级证据和缺口，不输出源码片段、生产账号、真实患者信息或真实生产验证结果。

## 发布负责人需要补齐的外部证据

以下事项不能由 PC 仓库源码自动证明，必须由发布负责人或对应系统负责人补齐：

1. 运行包一致性：确认正式发布使用的 PC 静态包、小程序包、后端运行包和 SQL 变更，与本地恢复材料和 GitHub commit 对应。
2. 微信正式发布：确认最新已认可小程序版本完成正式提审/发布，并且正式长期二维码已经切到新版本；体验二维码不能替代正式发布证据。
3. 备份：确认数据库、当前线上静态资源和后端运行包已有可恢复备份。
4. 回滚：确认出现问题时能回到上一个可用版本，并明确负责人和回滚窗口。
5. 审计和权限：确认谁能发布、谁能回滚、谁能查看日志；Codex 不执行生产部署。
6. 脱敏灰度记录：只保留脱敏记录，不提交真实账号、真实患者信息、真实 allow-list id 或生产验证结果。

## 发布负责人确认话术

发布负责人可以按下面几句话逐条确认，不需要提供真实账号或真实患者信息：

- 确认 PC 正式版一般情况表和当前用药保持只读，修改用药只走平板/小程序。
- 确认小程序正式发布版本以业务用户最后认可的小程序口径为准。
- 确认正式发布使用的后端运行包和 SQL 变更与恢复材料一致。
- 确认正式发布前已有数据库、前端静态资源和后端运行包备份。
- 确认出现问题时按发布负责人指定方案回滚。

以上确认完成后，代码可以作为正式上线评审材料的一部分；是否进入生产发布窗口仍由发布负责人按变更流程决定。
