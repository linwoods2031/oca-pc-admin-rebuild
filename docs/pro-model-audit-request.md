# Pro 模型项目审计请求

请以资深前端架构师、医疗评估系统 QA 和发布审计人的角度，审计这个 Vue 3 PC 后台重建项目。

## 项目背景

现有 PC 后台前端源码丢失，本项目在 `pc-rebuild` 中重建一套可维护的 Vue 3 后台，接入现有生产后端 `/prod-api`。要求不改生产后端、不改微信小程序、不直接覆盖线上生产静态目录。

注意：不要把真实账号、密码、测试患者姓名、身份证、手机号或生产验证细节提交到仓库；本文仅保留脱敏占位符。

## 重点审计范围

1. 功能完整性
   - 登录、患者列表、患者详情、新增、编辑
   - 一般情况调查表查看、编辑、新增、保存
   - 量表列表、量表明细、待填写量表保存、已提交只读保护
   - 总报告预览和 PDF 打印入口
   - 回访管理

2. API 和数据安全
   - 是否可能误写生产数据
   - 已提交量表只读保护是否可靠
   - 一般情况表新增/更新判断是否合理
   - token 和登录态处理是否有明显问题
   - `/prod-api` 代理和部署路径是否有风险

3. 代码质量
   - Vue 组件复杂度
   - 状态管理是否清晰
   - 错误处理是否会误导用户
   - 表单字段映射是否有明显遗漏
   - 是否存在后续维护困难点

4. 发布风险
   - `docs/deploy-plan.md` 灰度方案是否可执行
   - 是否存在覆盖生产目录风险
   - 回滚命令是否充分
   - 上线前还缺哪些自动化或接口验证

## 已完成验证

- `npm run build` 通过。
- 受控接口登录通过：`<username> / <password>`。
- 真实接口患者列表读取通过。
- 真实接口一般情况读取通过。
- 使用测试患者 `<test-patient>` 创建未提交测试评估，新增一般情况表返回成功。
- 使用同一测试评估读取 MMSE 量表明细，保存后量表得分刷新为 `33 / 正常`。
- 受控接口总报告 PDF：`/outpatient/check/composite/print/<outpatient-id>` 返回 `200 application/pdf`。

- 是否阻塞灰度部署。
- 灰度前必须补的验证。
- 可以灰度后再优化的事项。
- 对部署方案的修改建议。
