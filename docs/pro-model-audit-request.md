# Pro 模型项目审计请求

请以资深前端架构师、医疗评估系统 QA 和发布审计人的角度，审计这个 Vue 3 PC 后台重建项目。

## 项目背景

现有 PC 后台前端源码丢失，本项目在 `pc-rebuild` 中重建一套可维护的 Vue 3 后台，接入现有生产后端 `/prod-api`。要求不改生产后端、不改微信小程序、不直接覆盖线上生产静态目录。

注意：不要把真实账号、密码、测试患者姓名、身份证、手机号或生产验证细节提交到仓库；本文仅保留脱敏占位符。

## 重点审计范围

1. 功能完整性
   - 登录、患者列表、患者详情、新增、编辑
   - 一般情况调查表和当前用药只读查看
   - 量表列表、量表明细、待填写量表保存、已提交只读保护
   - 总报告预览和 PDF 打印入口
   - 回访管理

2. API 和数据安全
   - 是否可能误写生产数据
   - 已提交量表只读保护是否可靠
   - 一般情况表和当前用药是否在 PC 端保持只读
   - token 和登录态处理是否有明显问题
   - `/prod-api` 代理和部署路径是否有风险

3. 代码质量
   - Vue 组件复杂度
   - 状态管理是否清晰
   - 错误处理是否会误导用户
   - 表单字段映射是否有明显遗漏
   - 是否存在后续维护困难点

4. 发布风险
   - `docs/deploy-plan.md` 灰度方案是否可执行
   - 是否存在覆盖生产目录风险
   - 回滚命令是否充分
   - 上线前还缺哪些自动化或接口验证

## 自动化和外部证据口径

- 本地和 CI 证据以 `npm run verify`、`npm run release:evidence:verified` 和 GitHub Actions 为准。
- 文档和证据包不得包含真实账号、真实患者、真实 allow-list id 或生产验证细节。
- 业务用户已确认 PC 后台正式使用时一般情况表和当前用药只读，修改保留在平板/小程序流程中完成。
- 真实服务器运行包一致性、生产审批、备份、回滚和审计仍属于发布负责人外部证据。

## 请输出

- 按严重程度排序的问题清单，标明文件和原因。
- 是否阻塞灰度部署。
- 灰度前必须补的验证。
- 可以灰度后再优化的事项。
- 对部署方案的修改建议。
